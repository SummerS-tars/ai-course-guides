# 学习指南整合规范

> **用途**：统一维护 AI 课程学习指南的整合流程、质量下限与监修标准。适用于 `AI-Week*-学习指南.md`、`AI-PPT*-学习指南.md` 以及后续补充指南。  
> **工作流入口**：`.cursor/skills/ai-course-notebooklm/SKILL.md`  
> **Skill 详细规范**：`.cursor/skills/ai-course-notebooklm/docs/integration-guide.md`  
> **本文件定位**：面向实际写作/监修时的质量准则，避免规则散落在单个指南正文中。

---

## 1. 工作流底线

### 1.1 六阶段流程

```mermaid
flowchart LR
    P0["Phase 0<br/>资料盘点"] --> P1["Phase 1<br/>NotebookLM 采集"]
    P1 --> P15["Phase 1.5<br/>通读 raw + 知识图谱"]
    P15 --> P2["Phase 2<br/>按图谱写指南"]
    P2 --> P3["Phase 3<br/>叙事与可视化"]
    P3 --> P4["Phase 4<br/>用户迭代"]
    P4 --> P5["Phase 5<br/>定稿"]
```

| 阶段 | 必须产出 | 质量要求 |
|------|----------|----------|
| Phase 1.5 | `knowledge-graph.md` | 必须通读全部 raw，标出节点、叙事线、章节映射 |
| Phase 2 | 初稿指南 | 不能只粘贴 raw；必须按读者认知顺序重组 |
| Phase 3 | 叙事增强版 | 补全全景节、承接、追问、对比表、必要图示 |
| Phase 4 | Review 回写 | 用户指出的理解断点要沉淀为规范，而不只修当前句子 |

**禁止**：未通读 raw 就写指南；只把 raw 摘要拼接成正文；把通用监修经验写进单个学习指南正文。

### 1.2 raw / manifest 责任检查

当指南质量差时，先判断问题来源：

| 现象 | 更可能的问题 | 处理 |
|------|--------------|------|
| raw 没覆盖关键公式、例题、代码 | manifest / 采集问题 | 补采或重新设计 batch |
| raw 有公式和例题，但指南只写摘要 | 整合问题 | 回到 raw/周指南/PPT 指南补全 |
| 指南有术语表，但正文仍读不懂 | 正文解释不足 | 在首次使用处就地解释 |
| 代码示例有，但不知道每行体现什么 | 教学化不足 | 加逐行解释和运行流程 |

## 2. 认知编排原则

### 2.1 先「是什么 / 为什么」，再「怎么做」

每个大模块开始前必须回答：

| 问题 | 要求 |
|------|------|
| 这节解决什么问题？ | 一句话说明动机 |
| 学完能做什么？ | 3–5 条可检验能力 |
| 内部路线是什么？ | 问题链、流程图或小结 |
| 哪些是考试重点？ | 必考 / 高优先 / 了解 分层 |

不要直接进入公式、代码或术语表。

### 2.2 认知阶梯

```text
L0 定位与动机
  → L1 全景地图
  → L2 符号与概念
  → L3 机制 / 推导 / 代码
  → L4 数值或运行例子
  → L5 工程流程 / 考试模板
  → L6 跨周串联
```

整合顺序按读者理解顺序，不按 manifest 采集顺序。

### 2.3 三层叙事

- **章级叙事线**：大节开始写「本节要从 A 走到 B」。
- **节级问题**：每小节开头用 `> **本节要回答**：...`。
- **节间承接**：每节结尾说明「为什么下一节必要」。

## 3. 语言与解释标准

1. **名词不裸奔**：典故、算法名、英文术语、代码符号首次出现时，要交代场景、要解决的问题，以及它在本章中的作用。
2. **专业词首次白话解释**：先验、似然、稳态、阻尼因子、隶属度、Agenda、salience 等术语，第一次出现要翻成普通话。
3. **抽象困难要拆机制**：不能只说「复杂」「爆炸」「不稳定」。要说明什么数量变多、什么状态变化、为什么影响推理或训练。
4. **例子必须服务理解**：例子要明确解释哪个公式、机制或代码结构；有来源数值要保留关键计算，没有来源的数值不硬编。
5. **先建概念再给公式**：先讲变量/符号含义和问题背景，再给公式表；公式后解释每一项在流程中的位置。
6. **重点公式表不可被摘要吃掉**：CF、CNF、Resolution、BP、Viterbi 等手算/大题重点必须保留符号表、适用条件、步骤和例题。
7. **备考权重明确**：区分必考手算、概念理解、选择题、课堂补充，避免所有内容同等用力。
8. **正文解释不能依赖术语表兜底**：术语表可以预告概念，但正文首次真正使用核心词时仍要就地解释。

## 4. 系统 / 代码类指南最低结构

遇到 CLIPS 猜动物、Prolog 查询、Transformer、CNN、VAE、HMM/CRF 等系统或代码案例时，至少按下面顺序组织：

1. **系统全景**：它是什么、解决什么问题、用户/输入/输出是什么。
2. **核心特质**：为什么它能体现本章思想，例如前向推理、可解释、端到端训练、动态规划。
3. **核心对象**：有哪些主要数据结构、状态或事实，例如 `node`、`current-node`、`answer`、参数、隐状态。
4. **语法 / 符号入口**：解释最小语法单元或公式元素，例如 `assert`、`defrule`、LHS/RHS、`=>`、`?x`、`$?x`。
5. **运行流程**：按时间顺序说明状态如何变化，例如 Match → Agenda → Fire → 新事实。
6. **贴题示例**：示例必须逐行说明“这一行体现了哪个机制”。
7. **备考定位**：指出需要会读代码、会手算、会简答，还是了解即可。

### 4.1 代码符号解释要求

代码符号不能只出现在代码块中，正文必须说明：

| 维度 | 要回答 |
|------|--------|
| 角色 | 这个符号/语法在系统中负责什么？ |
| 位置 | 它写在模板、规则 LHS、RHS、公式还是命令里？ |
| 最小例子 | 给一个不依赖复杂上下文的小例子 |
| 常见误解 | 它是否容易和其它符号混淆？ |

例如 `assert` 要说明“向事实库添加新事实”；`=>` 要说明“分隔 LHS/RHS，不是 `;`”；`?c&brown|black` 要说明是槽内模式约束。

## 5. 示例与题目标准

### 5.1 示例必须贴合小节标题

小节标题说「复杂模式匹配」，示例就必须明确展示复杂性来自哪些语法，例如规则层 `or/not/exists`，槽内 `& | ~`，或 `multislot` / `$?`。不能只给一个相关但无法解释标题的问题。

### 5.2 大题 / 手算模板必须完整

对期末必考内容，指南至少包含：

- 符号表
- 公式表
- 适用条件
- 手算顺序
- 一个完整例题
- 易错点

例如 CF 不应只写“同向/异向合成”，还要有同正、同负、异号例子；Resolution 不应只给九步表，还要有 NL → FOL → CNF → 消解的完整模板。

## 6. 格式与渲染检查

定稿前检查：

- Markdown 表格列数是否一致。
- 表格内条件概率竖线是否写成 `\mid`，绝对值是否使用 `\lvert...\rvert` 或 `\lVert...\rVert`。
- LaTeX 是否有粘连，如 `\lambdaw`。
- Mermaid 标签是否含易坏字符；必要时改成纯文本。
- 代码块语言是否正确，如 `clips`、`python`、`mermaid`。
- 标题层级是否连续。
- 资料索引、raw 路径是否存在。

---

## 7. 监修检查清单

- 每个大板块是否先回答「本节要解决什么问题」。
- 首次出现的核心术语是否有中文、英文和白话解释。
- 抽象概念是否有机制拆解，而不是只给结论。
- 例子是否紧贴概念或公式，且没有无来源数值。
- 重点公式、符号表、适用条件和手算模板是否完整。
- 复习优先级是否与课堂和期末定位一致。
- 易混概念是否通过对比表或提醒块区分。
- 系统类内容是否先说明系统全貌、核心特质、输入输出和典型交互，再解释内部结构。
- 代码类内容是否解释了关键符号的角色、语法、使用位置和最小例子。
- 示例是否能直接回答本小节标题中的问题。
- 动态流程是否说明了状态如何变化，例如事实如何被 `assert/retract/modify`，规则为何进入 Agenda，节点如何跳转。
- 术语表中的关键名词是否也在正文首次使用处得到足够解释。
- Markdown 表格、公式、mermaid、路径引用是否能正常渲染。
