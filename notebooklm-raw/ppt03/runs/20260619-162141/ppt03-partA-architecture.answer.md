根据 **课件 03 CLIPS** 以及 **Week 15 课程记录**，产生式系统（Production System）的架构、运行周期及议程（Agenda）管理机制详解如下：

### 1. 产生式系统三组件及其职责

产生式系统是符号主义专家系统的核心，其模块化结构主要由以下三部分组成：

*   **事实库 (Working Memory / Fact list)**：
    *   **职责**：存储当前已知的所有事实及关于世界状态的描述 [1, 2]。
    *   **说明**：它相当于系统的“工作内存”，反映了推理过程中的当前状态。事实可以通过 `deftemplate` 定义结构，或以有序事实（Ordered facts）形式存在 [3, 4]。
*   **知识库 (Knowledge Base / Rules)**：
    *   **职责**：存储所有的产生式规则，即 **“如果……那么……” (IF-THEN)** 规则 [1, 2]。
    *   **说明**：每条规则由左手边（LHS，前提条件）和右手边（RHS，动作）组成。它代表了领域专家的经验知识 [2, 5]。
*   **推理引擎 (Inference Engine)**：
    *   **职责**：作为系统的控制器或解释器，负责驱动整个推理过程 [1, 2]。
    *   **说明**：它内置于编程语言中，通过执行“识别-动作周期”来决定何时以及如何执行知识库中的规则 [2]。

---

### 2. 识别-动作周期 (Recognize-act cycle) 逐步流程

推理过程是一个循环往复的周期，其逐步流程如下（基于伪代码逻辑与课程记录）：

```ascii
[ 初始化：将初始事实输入事实库 ]
       |
       v
+-----------------------+
| 1. 匹配 (Match)       |<---------------------------+
|    检查事实库是否满足 |                            |
|    规则的 LHS 模式    |                            |
+-----------------------+                            |
       |                                             |
       v                                             |
+-----------------------+                            |
| 2. 激活 (Activation)  |                            |
|    满足条件的规则进入 |                            |
|    议程 (Agenda)      |                            |
+-----------------------+                            |
       |                                             |
       v                                             |
+-----------------------+                            |
| 3. 冲突消解          |                            |
|    (Conflict Resolution)|                          |
|    从议程中选出优先级 |                            |
|    最高的激活项       |                            |
+-----------------------+                            |
       |                                             |
       v                                             |
+-----------------------+      +------------------+  |
| 4. 执行 (Act)         |----> | 5. 更新事实库     |--+
|    顺序执行 RHS 动作  |      | (添加/删除/修改) |
+-----------------------+      +------------------+
       |
       v
[ 6. 停止检查：若议程为空或执行了 Halt/Break 则结束 ]
```

**步骤详细说明 [2, 6]**：
1.  **匹配 (Match)**：系统通过 **Rete 算法 [7, 8]** 高效检查哪些规则的前提条件（LHS）被当前事实库满足。
2.  **激活 (Activation)**：被满足的规则实例（Activation）会被放入议程。
3.  **冲突消解 (Conflict Resolution)**：如果议程中有多个激活项，推理引擎根据预设逻辑选出一个执行。
4.  **执行 (Act)**：顺序执行选中规则 RHS 上的所有动作。
5.  **更新**：执行动作后，事实库内容会发生变化，这可能触发新的规则匹配，或使议程中原有的激活项失效（移除）[6]。

---

### 3. Agenda (议程) 的排序与规则触发

**议程 (Agenda)** 是推理引擎内部维护的一个列表，存放所有已激活但尚未执行的规则实例 [2]。

*   **排序机制 (Priority / Salience)**：
    *   **显着性 (Salience)**：CLIPS 使用 `salience` 来指定规则的优先级 [9]。
    *   **取值范围**：范围在 -10,000 到 10,000 之间，默认值为 0 [9]。
    *   **排序原则**：推理引擎在 **冲突消解 (Conflict Resolution)** 阶段，会优先选择 **显着性 (Salience)** 值最高（即优先级最高）的激活项进行 fire（触发执行）[6, 9]。
*   **规则的触发与移除 [2, 6]**：
    *   **触发执行**：一旦规则被选中并 fire，它的 RHS 动作被执行，该激活项随即从议程中移除。
    *   **动态更新**：
        *   若新事实的加入使其他规则的 LHS 得到满足，这些规则会产生新的 **激活项 (Activations)** 加入议程。
        *   若 RHS 动作删除了某些事实，导致议程中现有规则的 LHS 不再被满足，对应的激活项也会被 **自动清除 (Remove activations)**。
*   **控制模式 [10, 11]**：
    *   除了显着性，还可以通过 **控制模式 (Control pattern)**（如在事实中设置 `phase` 字段）或 **模块化 (defmodule)** 机制来管理规则执行的顺序和焦点 [10, 11]。
